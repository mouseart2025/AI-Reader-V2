# B 站视频脚本 v2（终稿）

**时长**：6-7 分钟 | **风格**：屏幕录制 + 画外音 | **不出镜**

---

## 【0:00-0:15】钩子——3 秒定生死

**画面**：凡人修仙传关系图谱全貌，密密麻麻的红色线条，缓慢推进

**旁白**：
> "2480 个人物，4484 条关系，一本 2000 多章、700 万字的小说。这张图上每一个点、每一条线，全是 AI 自动读完后提取出来的。而做这个工具的我——全程没写一行代码。"

**画面字幕**：「0 行代码」大字弹出 + 「3 周」

---

## 【0:15-1:00】起因（45 秒）

**画面**：起点/番茄阅读 app 的超长章节列表滚动（模糊处理避免版权），或者直接用 AI Reader 书架页上的凡人修仙传

**旁白**：
> "追长篇网文的人应该都有这个痛苦。《凡人修仙传》，2000 多章，放下几个月再拿起来——这人谁来着？南宫婉和那个妖女什么关系？这个宗门之前出现过吗？
>
> 我就想，这事 AI 应该能干。让它逐章读完整本小说，把人物关系、地点、势力、事件全部提取出来。于是春节期间我给自己定了个挑战——用 Claude Code，不写一行代码，从零做一个小说分析系统。"

---

## 【1:00-1:50】翻车与转折（50 秒）

**画面**：文章中那张早期"聊天窗口获取世界地图层级树"的截图

**旁白**：
> "第一版还真跑起来了——React 前端、Python 后端，都有。但用起来就不对了，功能东一块西一块，不像个产品。原因很简单——我需求就是散的，AI 做出来的东西自然也散。
>
> 你看，最早的版本连 UI 都没有，我是通过聊天窗口查看分析结果的。"

**画面**：切到文章中 BMAD 的 Excalidraw UX 设计稿截图

**旁白**：
> "后来我找到了一个叫 BMAD 的 AI 敏捷开发框架，这是整个项目的转折点。它让我从一个提需求的人，变成了一个产品经理——先写 PRD，再做交互设计，最后才开发。这是 AI 帮我出的交互设计稿。
>
> 突然之间我找到感觉了——我不是在写代码，我是在管一个 AI 团队。春节假期高频迭代了 30 多个版本，就做出了现在这个。"

---

## 【1:50-4:20】功能展示（2 分半，核心段落）

**节奏提示**：这段全部是屏幕录制，每个功能控制在 20-30 秒，BGM 可以在这里稍微提能量。

### 功能 1：上传分析（20 秒）

**画面**：文章配图——封神演义导入面板 → 费用预估弹窗

**旁白**：
> "使用很简单。扔一本 TXT 进去，自动拆分章节。点分析，AI 就开始逐章阅读。用云端模型的话，还会提前告诉你大概花多少钱。"

### 功能 2：人物关系图谱（40 秒）⭐ 最重头

**画面**：
1. 西游记图谱全貌 → 鼠标拖动节点 → 拉动章节滑块看关系变化
2. 点击孙悟空 → 弹出人物卡片 → 展示别名列表（行者/齐天大圣/美猴王...）
3. 切到凡人修仙传的超大规模图谱

**旁白**：
> "这是最有冲击力的——人物关系图谱。70 多种关系类型，6 种颜色分类。拖动滑块可以看不同章节范围的关系演变。
>
> 点开人物卡片，你能看到 AI 记录的所有信息——别名、外貌、关系、能力、足迹、经历。西游记里孙悟空有十几种称呼，行者、齐天大圣、美猴王、老孙……系统能自动合并成同一个人。这个功能我调了很多轮才搞定。
>
> 凡人修仙传更夸张——2480 个人物，你看这满屏的红线，全是敌对关系，完美反映了这本小说弱肉强食的主基调。"

### 功能 3：世界地图（30 秒）⭐ 视觉亮点

**画面**：
1. 文章配图——西游记手绘幻想地图
2. 神秘岛真实地理坐标地图
3. 平凡的世界的混合地图
4. 播放轨迹动画（如果能录到）

**旁白**：
> "然后是世界地图。AI 从文本提取地名和空间关系，自动生成手绘风格地图，有河流、群落、地形，还能播放角色轨迹动画。
>
> 现实背景的小说会自动匹配真实坐标——像《神秘岛》直接用的真实世界地图。半虚构的像《平凡的世界》，也能画出大概的地理关系。
>
> 说实话这是我花时间最多的功能，离我想要的游戏地图效果还有差距，但已经够用了。"

### 功能 4：时间线 + 百科 + 问答（30 秒）快速切

**画面**：时间线截图 → 百科 20644 个实体的截图 → 问答界面

**旁白**：
> "时间线按章节展示事件，支持按人物和类型筛选。百科自动生成每个实体的档案——凡人修仙传提取出了 2 万多个实体。智能问答基于原文回答，不是瞎编，所有答案都能溯源到具体章节。"

### 功能 5：阅读 + 剧本模式（15 秒）

**画面**：文章配图——阅读页实体高亮 + 右侧剧本面板

**旁白**：
> "阅读时实体自动高亮，5 种颜色区分人物、地点、物品。还有剧本模式，按场景拆分，方便做影视改编参考。"

### 功能 6：技术栈一嘴带过（15 秒）

**画面**：设置页 AI 引擎面板 → MiniMax 性能测试截图

**旁白**：
> "技术上简单说——前端 React，后端 Python FastAPI，桌面应用 Tauri 打包，Mac 和 Windows 都有。LLM 支持 10 家供应商，本地 Ollama 免费但慢，我现在主力用 MiniMax，分析一章两三分钟，也不贵。数据全在你本地，不上传到任何服务器。"

---

## 【4:20-4:50】AI 代劳到什么程度（30 秒）

**画面**：ai-reader.cc 官网 → GitHub Actions 构建页面 → 文档页面

**旁白**：
> "除了产品本身，还有些东西也是 AI 做的——官网、CI/CD 流水线、产品文档，甚至包括这次营销文案的初稿。我的角色更像导演，把控方向、定义标准、验收结果。代码一行没写，但每天都在'工作'。"

---

## 【4:50-5:50】踩坑（真诚段，60 秒）

**画面**：可以配一些"修 bug"的意象——比如图谱上一堆乱连的线（别名没合好的状态），或者地图上标红的冲突点

**旁白**：
> "不想拍成 AI 吹捧视频，说说真实踩过的坑。
>
> **第一，别名合并。** 西游记里'妖精''那怪''泼猴'这些泛称，如果不过滤，会把不相关的角色绑在一起。修仙小说更夸张，'前辈''道友''师兄'满天飞，一个组能膨胀到几十个成员。这个问题前后花了很多轮迭代才搞定。
>
> **第二，地图质量。** 小说里的空间描写经常是模糊甚至矛盾的——作者写到后面自己都忘了前面怎么说的。AI 提取出来的数据需要大量后处理。
>
> **第三，也是最重要的——需求不清楚等于废代码。** 你只说'帮我做个小说分析工具'，AI 做出来的东西真的很泛。必须把需求拆细，说清楚交互逻辑和验收标准。**AI 的产出质量上限，就是你需求的质量。**"

**画面字幕**：重复强调「AI 的产出上限 = 你需求的质量」

---

## 【5:50-6:30】结尾（40 秒）

**画面**：各功能页快速蒙太奇切换（图谱 → 地图 → 时间线 → 百科 → 问答 → 阅读）→ 定格在官网

**旁白**：
> "这个项目让我真切感受到——AI 时代，把想法变成产品的门槛确实大幅降低了。但降低不等于消失，你仍然需要想清楚做什么、为谁做、做成什么样。
>
> 如果你也有一个想做的东西，不管会不会写代码，不妨试试。核心就一句话：把需求结构化地表达清楚，然后交给 AI 执行。
>
> 项目完全开源，官网有在线 Demo 可以直接体验，链接在评论区。觉得有用的话一键三连，后续可能出技术拆解的详细视频。"

**结尾画面（3 秒）**：
```
ai-reader.cc
github.com/mouseart2025/AI-Reader-V2
⭐ 一键三连 ⭐
```

### B 站发布配置

- **标题方案**（二选一）：
  - A：`AI 读完 2000 章凡人修仙传，画出了这张图`（好奇心驱动，推荐）
  - B：`不写一行代码，我用 AI 做了个小说分析神器`（和知乎统一）
- **封面**：凡人修仙传图谱全景 + 大字"2480 个人物 4484 条关系"
- **分区**：知识 → 科技科普（比软件应用流量大）
- **标签**：AI、Claude、独立开发、凡人修仙传、网文、小说、人物关系
- **置顶评论**：Demo 链接 + GitHub 链接 + "想看哪本小说的分析？评论区告诉我"
- **简介**：直接复制知乎文章开头 + 链接

### 和知乎联动

在知乎文章末尾追加一行：
> "B 站有视频版，更直观：[链接]"

知乎版侧重文字深度，B 站版侧重视觉冲击，互相导流。
